7 Principles of Software Testing

Ch.1
PEED PCN

⚠️ For scenario-based questions, only explain principles directly relevant to the scenario.

P — Testing shows the Presence of defects, not their absence

Testing can show the product fails but cannot prove 100% defect-free. It only reduces the probability of hidden defects remaining.

E — Exhaustive testing is impossible

Testing all combinations of inputs and preconditions is not feasible. Testers must use risk assessment and priorities to control test efforts.

E — Testing activities should start as Early as possible

Early testing detects defects sooner, reducing the cost and time required to fix them later in the lifecycle.

D — Defect clustering

Defects are not evenly distributed — they cluster together. A small number of modules usually contain most defects. If many defects are found in one area, there are usually more in that area.

P — The Pesticide paradox

The same test cases repeated over and over will eventually stop finding new bugs. Test cases must be regularly reviewed and modified/updated.

C — Testing is Context-dependent

No two systems should be tested in the exact same way. Testing must be adapted to the risks and environment of the application (e.g., safety-critical vs e-commerce).

N — No failures means the system is useful is a fallacy

Finding failures and repairing defects does not guarantee the system meets users' expectations and needs.

Software Error → Defect → Failure

Ch.1

1. Software Error

  • • A human mistake made by a person
  • • Logical mistake in code
  • • Anyone in the team can make mistakes during different phases of development

2. Software Defect (Bug)

  • • Difference between expected and actual results found by a tester
  • • Reflects inability or inefficiency to comply with specified requirements

3. Software Failure

  • • Observable incorrect behaviour of the system when run in real environment
  • • Occurs when the software fails to perform in the real environment
Error → Defect → Failure

Software Testing Life Cycle (STLC)

Ch.1
RPD SEC

A planned and systematic approach of testing software to ensure its quality, functionality, and security.

1. Requirement Analysis

Analyze project requirements for testing. Includes formal Team Review.

2. Test Planning

Creating a strategy for testing / test plan.

3. Test Case Development

Design test cases based on requirements.

4. Test Environment Setup

Involves Test Data Generation. Configure hardware, software, network and test bed for testing.

5. Test Execution

Running test cases to find defects. Involves Test Monitoring & Control via Test Logs.

6. Test Cycle Closure

Final analysis and documentation. Generate Test Summary Report.

Incremental Integration — Advantages

1. Earlier Error Detection

Interface mismatches and incorrect assumptions are caught sooner.

2. Easier Debugging

Issues are easier to isolate because they are typically found in the most recently added module.

3. More Thorough Testing

Modules receive more exposure as they are exercised repeatedly as new modules are added.

Stubs vs Drivers

Ch.2

Stub

  • • Dummy section of code used in Top-Down testing
  • • Simulates a sub-module that hasn't been developed yet
  • • Replaces a called/lower-level module

Driver

  • • Dummy program used in Bottom-Up testing
  • • Simulates the main / calling module
  • • Triggers the module under test

Bug Classifications

Ch.6

1. Requirements Failure

The system functions, but it does not behave according to the client's documented business rules or specifications.

2. Non-Requirements Failure

Technical defects, coding errors, or design flaws that violate standard software behavior, even if not explicitly written in the business requirements (e.g., crashes, UI glitches).

3. External Failure

Failures caused by interactions with third-party software, external hardware, or external APIs outside the direct control of the system.

6 Main Components of the SQA System

Ch.9

An SQA system is an entire organizational framework — not just testing. It comprises six classes of components.

A. Pre-Project Components

Taken before actual development begins. Ensures project commitments, resources, schedules, and budgets are adequately defined.

Contract Review Development & Quality Plans

B. Software Project Life Cycle Components

Core quality tasks during actual development phases (design, coding, testing) and operation-maintenance stage.

Reviews (FDR, Peer Reviews) Expert Opinions Software Testing (Black/White Box)

C. Infrastructure Components (Error Prevention & Improvement)

Foundational tools and guidelines used globally to prevent faults and improve productivity.

Procedures & Work Instructions Templates & Checklists Staff Training & Certification Preventive & Corrective Actions Configuration Management

D. Management SQA Components

Supports managerial control to prevent schedule and budget failures.

Project Progress Control Software Quality Metrics Software Quality Costs

E. SQA Standards, System Certification & Assessment Components

Ensures the organization's processes align with the rest of the industry and international standards.

ISO 9000-3 IEEE Standards CMM / CMMI

F. Human Components (Organizational Base)

The actual people who make up the organizational base of SQA. Includes managers, testing personnel, SQA Unit, and SQA practitioners.

  • • Develop and support implementation of SQA components
  • • Actively detect deviations from SQA procedures and methodologies
  • • Constantly suggest improvements to the SQA system

Factors that decide the SQA System used:

Organizational: Size of organization, range of products, type of software clientele
Project & Maintenance: Level of complexity, staff experience, extent of code reuse
Professional Staff: Professional qualifications, training levels, teamwork history
A

SQA Activities

Ch.9

Part 1: Core SQA Activities in Testing PIE OER

1. Plan the Testing and SQA Processes

Before any testing begins, establish a clear plan that dictates what will be tested, when, how, and by whom. Create quality management plan, test plan and test case.

2. Implementation of Test-Oriented Management

Adopt specific development methodologies to integrate quality from the start:

  • • Extreme Programming (XP): Agile framework to improve software quality and responsiveness to changing requirements
  • • Test-Driven Development (TDD): "Test-First Approach" — test cases are written before any actual code implementation
  • • Pair Programming: Two developers work together at one workstation (one writes, one reviews each line) to produce higher-quality software

3. Ensure a Suitable Work Environment

Provide the SQA and testing teams with proper tools, environments, and conditions. Involve dedicated SQA team from the beginning to start testing early.

4. Optimize the Use of Automated Tests

Maximize test automation to increase efficiency, reduce manual repetitive tasks, and ensure consistent execution of regression tests.

5. Employ Code Quality Measurements

Track measurable metrics: Rate of Delivery, Testability, Product Usability, Reliability, Security, Maintainability, Performance Efficiency.

6. Report Bugs Effectively

A good bug report makes testing more efficient by clearly identifying the problem and navigating engineers toward a solution. A badly written report can lead to serious misunderstandings and wasted time.

Part 2: Broad SQA Practices

Reviews & Inspections — Reviewing requirements, test plans, test cases, source code
Test Process Monitoring — Monitoring all testing activities against defined plans
Quality Audits — Independent audits to verify compliance with quality standards
Defect Tracking & Management — Recording, prioritizing, tracking, and resolving defects
Metrics & Reporting — Collecting and analyzing quantitative testing data

✅ Benefits

  • • Ensures ultimate quality of both process and final product
  • • Standardizes procedures and formatting of artifacts across the organization
  • • Enhances traceability (ability to trace a defect back to its original requirement)

⚠️ Challenges

  • • Resource Heavy: Requires more people, tools, and environments
  • • Requirement Issues: Unclear or changing requirements can derail SQA plans
  • • Time Constraints: Tight timelines make thorough SQA difficult
  • • Budget: Insufficient funds allocated to QA
  • • Human Factor: Resistance from developers or management to adapt to QA processes

Defect Management

Ch.9

Defect Reporting & Meeting

  • • Defect Reporting: Written description of defects in software
  • • Defect Meeting: Best way to disseminate info among testers, analysts, and developers
  • • Meeting is conducted at end of every day
  • • Defects are categorized by type and severity

Defect Classification

  • • Helps test planning and prioritization
  • • Classification may vary depending upon requirements of project

Showstopper (X)

  • • Impact of the defect is severe
  • • System cannot be tested without resolving; internal solution may not be available
  • • High Priority

Critical (C)

  • • Impact is severe, BUT an internal solution (workaround) is available
  • • Medium Priority

Non-Critical (N)

  • • All defects not in X and C categories
  • • Can be solved via documentation or user training
  • • Minor issues — does not affect testing
  • • Low Priority

Defect Priority

High Priority

Further development or testing can't occur until defect is repaired.

Medium Priority

Defect needs to be solved as soon as possible; it is impairing development and testing.

Low Priority

Can be repaired after more serious defects are fixed.

Test Reports (IEEE 829)

Ch.9

Test Incident Report

  • • Written description of an incident (variation/deviation observed from expected system behavior) observed during testing
  • • An incident is basically any unexpected behavior or response from software that requires investigation (broader than just defect)
IEEE 829 Template includes:
  • • Test incident report identifier
  • • Summary (references test procedure & test logs)
  • • Incident description (inputs/expected, actual results, anomalies, procedure step)
  • • Impact (actual damage + severity/priority assessment)

Test Summary Report

  • • Created at a milestone or at the end of a test level
  • • Describes the results of a given level or phase of testing
IEEE 829 Template includes:
  • • Test summary report identifier
  • • Summary
  • • Variances
  • • Comprehensive assessment
  • • Summary of results
  • • Evaluations
  • • Summary of activities
  • • Approvals

IEEE 829 Test Summary Report — Detailed Sections

1. Test Summary Report Identifier — Unique ID to identify this summary report and indicate the level of testing.
2. Summary
Identify all relevant support materials so the reader knows which version and release is being reported on. Identify specific version of external packages. Includes: test items, environment, references.
3. Variances
Document any changes or deviations from agreed-upon areas in the reference documents. Include supporting documents for the deviation/change.
4. Comprehensive Assessment
Evaluation of testing and test process in terms of documented test objectives. Includes: evaluation of test coverage (total objectives, objectives covered/omitted), identification of uncovered attributes, surprising trends and test process changes.
5. Summary of Results
Report on overall status of incidents (good and bad). Includes: total incidents (by severity/priority, by cost/failure impact), defect patterns, open/unresolved incidents.
6. Evaluations
Objective assessment of failure likelihood and overall quality against criteria in the test plan. Includes: limitations (incomplete/dropped features), failure likelihood (high/medium risk areas, good quality areas).
7. Summary of Activities
Cover planned activities and changes. Includes: staff time used (hours/day/week), elapsed vs staff time, costs (planned vs actual), variances and reasons (scope changes, requirements changes, defect trends, loss of personnel, test environment issues).
8. Approvals
List and signature block for each approving authority. Must match the test plan.

McCall's Software Quality Factors

Ch.9

McCall's Classic Quality Factors Model — 3 categories, 11 total factors.

1. Product Operation Factors (Daily Use)

Correctness: Does it do what I want? — Ensures software meets exact specifications and provides required outputs accurately.
Reliability: Does it do it accurately all the time? — Deals with failure rate and ability to provide uninterrupted service.
Efficiency: Will it run on my hardware as fast as it needs to? — Concerns hardware resources (processing power, memory, storage) needed to perform functions.
Integrity (Security): Is it secure? — Focuses on authorization, access levels, and protecting information against unauthorized access.
Usability: Can I run it easily? — Measures scope of staff resources needed to train a new employee and operate the system.

2. Product Revision Factors (Fixing/Changing — Maintenance)

Maintainability: Can I fix it easily? — Effort to identify root causes of failures, correct errors, and verify corrections.
Flexibility: Can I change it easily? — Capabilities and effort for adaptive and perfective maintenance (adding features, modifying for new business rules).
Testability: Can I test it easily? — Includes special diagnostic features embedded in programs for automatic checks to find faults easily.

3. Product Transition Factors (Moving/Sharing)

Portability: Will I be able to use it on another machine? — Effort required to transfer from one hardware/software environment to another.
Reusability: Will I be able to reuse some of the software? — Extent to which a program or modules can be reused in other applications.
Interoperability: Will I be able to interface it with another system? — Effort required to couple one system with another (e.g., HR software with Accounting software).

Software Maintenance Types

Ch.9

1. Corrective Maintenance

The correction of software faults/failures.

2. Adaptive Maintenance

Adapting the current software to additional circumstances and new customers without changing the core software itself. (e.g., moving to new OS)

3. Perfective Maintenance

Enhancement and improvement of existing software with respect to locally limited issues. (Adding new features, improving functionality)

SQA Metrics — Maintenance Phase (McCall's Product Revision)

Maintainability Metrics:

  • • Time to Fix: How quickly can a bug be traced and resolved?
  • • Code Modularity: Does changing one module break another?
  • • Documentation Quality: Is code well-commented?

Flexibility Metrics:

  • • Man-hours to implement a change request without rewriting core architecture

Testability Metrics:

  • • Does the software include embedded diagnostic features for automatic checks?
  • • High testability = easier fault finding during maintenance

Software Quality Control (SQC) Overview

Ch.10

Primary Objective

Find problems and fix mistakes in the software. Ultimate objective is the withholding of any product that does not qualify. Uses process measurement and data analysis to guarantee quality.

Timing of Execution

SQC is a reactive measure. Inspections, testing, and QC activities take place as development is completed, but strictly before the product is shipped to the client.

Approach to Errors

Unlike QA (which prevents causes of errors), SQC accepts the product is already built and focuses on detecting errors that slipped through. Involves hands-on execution and checking (dynamic testing, identifying bugs, reporting failures).

Relationship to QA

Quality Control is actually a subset (part) of Quality Assurance. SQA provides the overall framework and standards; SQC executes the tests to ensure those standards were met.

Product Metrics (Customer Service Performance)

Help Desk (HD) Services Metrics

  • • Measures support provided to users — instructing on usage and solving implementation problems
  • • Data Source: customer calls and support tickets
  • • High metric = poor Product Usability or poor user manuals (software too hard to use)

Corrective Maintenance Services Metrics

  • • Measures actual correction of software failures identified by customers or detected by service team
  • • Data Source: failure reports and bug tracking systems
  • • High metric = poor code quality, low reliability, inadequate testing during development

Total Quality Management (TQM) Principles

Ch.11
CE PI SC FC

TQM Implementation Steps

  1. Top management learns about and decides to commit to TQM — identify it as organizational strategy
  2. Organization assesses its current culture, customer satisfaction, and quality management systems
  3. Top management identifies and communicates core values and principles
  4. TQM master plan is developed based on the initial assessments
  5. Management forms teams for process improvement efforts and maps critical processes

8 TQM Principles

1. Customer-focused (C)

The customer ultimately determines the level of quality. No matter how much an organization invests in training, design, or new tools, the effort is only worthwhile if the customer finds value in the final product or service.

2. Total Employee Involvement (E)

All employees must participate in working toward common goals. Management is responsible for providing a proper, empowering environment that removes fear from the workplace.

3. Process-centered (P)

Focuses heavily on "process thinking." A process is a series of steps that transform inputs from suppliers into outputs for customers. Performance must be continuously monitored to detect unexpected variations.

4. Integrated System (I)

Focuses on horizontal processes that interconnect different vertical functional departments. Everyone must clearly understand the organization's vision, mission, and quality policies.

5. Strategic and Systematic Approach (S)

Quality cannot be an afterthought; it must be managed. Involves formulating a strategic plan that integrates quality as a core, foundational component of the organization's overarching goals.

6. Continual Improvement (C)

A major thrust of TQM. Drives an organization to constantly be analytical and creative in finding new ways to become more competitive, effective, and efficient.

7. Fact-based Decision Making (F)

Decisions should never be based on guesses. Requires continuous collection and analysis of performance data and history to improve decision accuracy, achieve consensus, and make predictions.

8. Communications (C)

Effective communication (involving strategy, method, and timeliness) is vital. It plays a massive role in maintaining high morale and motivating employees at all levels, especially during times of organizational change.

CMM Maturity Levels

Level 1

Initial

Processes are unpredictable, disorganized, and ad hoc. Success depends on individual heroics rather than defined processes. No KPAs required.

Example: Programmers code the ERP system directly without any formal requirement documents or testing plans.

Level 2

Repeatable

Basic project management techniques established. Successful processes are documented so they can be repeated on similar projects.

KPAs: Requirements Management, Software Project Planning, SQA, Configuration Management.

Example: Team uses a tool like Jira to track basic bugs and schedules.

Level 3

Defined

Organization develops its own standard software development methodology. Processes are standardized and integrated across the organization.

KPAs: Peer Reviews, Inter-group Coordination, Training Program, Organization Process Definition.

Example: Every team follows the exact same formal SDLC manual, using standardized templates.

Level 4

Managed (CMM) / Quantitatively Managed (CMMI)

Organization monitors and controls its processes through quantitative data collection and analysis.

KPAs: Quantitative Process Management, Software Quality Management.

Example: Management tracks precise defect density metrics (e.g., 2 bugs per 1000 lines of code) to predict system reliability.

Level 5

Optimizing

Focus is on continuous process improvement. Innovative functionality and feedback loops are used to prevent defects and improve processes.

KPAs: Defect Prevention, Technology Change Management, Process Change Management.

Example: Organization analyzes historical defect data and implements an AI-driven automated testing tool to permanently prevent specific types of defects.
